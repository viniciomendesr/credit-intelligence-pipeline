Fase 5 · Modelagem ML + Benchmarking contra Baseline Rule-Based
"Até aqui, a classificação de risco era uma regra SQL de 3 linhas. Quanto um modelo treinado no mesmo mart realmente ganha contra isso? E como você versiona esse modelo em produção?"

Entrega: src/train.py que treina um XGBoost no mart pra prever defaulted, calcula AUC + precision@k + log-loss, compara contra o baseline rule-based (risk_tier == 'HIGH'), salva models/model.pkl calibrado e reports/model_metrics.json com a comparação, e sobe o artefato pro bucket GCS.

Pré-requisito: mart_credit_features.parquet gerado (Fase 2) com coluna defaulted como label. Bucket GCS criado (ver Fase 4.3). Libs: scikit-learn, xgboost, joblib.

Por que esta fase existe? Até aqui o projeto tinha "inteligência" apenas nominalmente — a classificação de risco vinha de uma regra SQL de 3 linhas. Essa fase finalmente usa o label real do Kaggle (a coluna defaulted) como target de um modelo supervisionado e responde, com números concretos, a pergunta clássica: "quanto um modelo ganha vs uma regra boba?". O artefato resultante (model.pkl no GCS) é o que o Bônus Fase 5 vai consumir pra trocar a heurística de desvio-vs-mediana por SHAP values principiados.
Desafio 5.1 — Preparação do dataset + split estratificado · arquivo: src/train.py
📥 Entrada: mart_credit_features.parquet 📤 Saída: X_train, X_test, y_train, y_test em memória 🔗 Usado por: 5.2 (fit) e 5.3 (avaliação)
🎯 O que construir

Função prepare_dataset(mart_path) em src/train.py que carrega o mart, seleciona features, e faz split 80/20 estratificado por defaulted.

  • Features (X): tudo do mart exceto applicant_id (chave), defaulted (label), risk_tier (vazamento — deriva da regra que queremos bater), loaded_at (metadado), e has_90day_default (vazamento direto do label)
  • Target (y): coluna defaulted — binária
  • Split: train_test_split(X, y, test_size=0.2, stratify=y, random_state=42) — estratificação é crítica porque o dataset é desbalanceado (~6.76% positivos)
  • Dtype: garantir que todas features sejam numéricas — XGBoost não aceita object/string sem transformar
Por que excluir risk_tier das features? A regra SQL da Fase 2 derivou risk_tier a partir de revolving_utilization, late_30_59_days, e late_90_days. Se a gente alimenta risk_tier como feature, o modelo só tem que "reproduzir a regra" — não consegue descobrir padrão melhor. O propósito desta fase é descobrir um padrão melhor, não memorizar o existente. Excluir a coluna força o modelo a reaprender as interações a partir das features primárias.
✅ Como testar

(a) y_train.mean() e y_test.mean() devem ser ~0.068 (estratificação funcionando). (b) Número de features em X_train.columns é igual a número de colunas do mart menos 5 (as excluídas). (c) Sem NaN em X_train.dtypes — todas numéricas.

🎯 Expectativa de resultado

Train set ~115k linhas, test set ~29k linhas, ~6.8% positivos em ambos, ~11 features numéricas. Rodar duas vezes com mesmo random_state produz split idêntico — reprodutibilidade garantida.

📖 Referência — só abra após sua tentativa 
import pandas as pd
from sklearn.model_selection import train_test_split

EXCLUDE_COLS = {"applicant_id", "defaulted", "risk_tier", "loaded_at", "has_90day_default"}

def prepare_dataset(mart_path: str = "data/marts/mart_credit_features.parquet"):
    df = pd.read_parquet(mart_path)
    feature_cols = [c for c in df.columns if c not in EXCLUDE_COLS]
    X = df[feature_cols].copy()
    y = df["defaulted"].astype(int)
    return train_test_split(X, y, test_size=0.2, stratify=y, random_state=42)
Desafio 5.2 — Treinar XGBoost + calibração de probabilidade · arquivo: src/train.py
📥 Entrada: X_train, y_train 📤 Saída: models/model.pkl calibrado + metadados 🔗 Consumido por: 5.3 (métricas) + Bônus F5 (SHAP)
🎯 O que construir

Função train_model(X_train, y_train) que treina um XGBoost classifier, aplica calibração de probabilidade via CalibratedClassifierCV, e salva o resultado em models/model.pkl junto com metadados (features, data, version).

  • Classifier base: XGBClassifier(n_estimators=200, max_depth=5, learning_rate=0.1, scale_pos_weight=~13) — scale_pos_weight compensa o desbalanceamento (negatives/positives ≈ 13)
  • Calibração: CalibratedClassifierCV(xgb, method='isotonic', cv=3) — XGBoost por default retorna probabilidades mal calibradas; isotonic regression ajusta pra predict_proba refletir frequência real
  • Persistência: joblib.dump({'model': model, 'features': feature_cols, 'trained_at': iso_now, 'git_sha': sha}) — guardar as features junto evita ordem trocada em produção
Por que calibrar? XGBoost retorna scores que ordenam bem (ranking correto) mas não são probabilidades reais — um score de 0.8 não significa "80% chance de default". Calibração faz essa correção: depois da calibração, de 100 casos com score 0.8, aproximadamente 80 vão inadimplir. Pra narrativa do Bônus F5 ("sua probabilidade estimada é de 34%"), calibração é pré-requisito. Sem isso, a narrativa mente.
✅ Como testar

(a) model.predict_proba(X_test)[:, 1].mean() deve ficar próximo de y_test.mean() (≈0.068). (b) model.predict_proba(X_test) retorna valores em [0, 1] — sem negativos ou >1. (c) joblib.load("models/model.pkl") recarrega e roda predict sem warnings de versão.

🎯 Expectativa de resultado

Modelo serializado em ~5-10 MB. Tempo de fit local: 30s-2min dependendo do hardware. Predict em 29k linhas <1s.

📖 Referência — só abra após sua tentativa 
import joblib, subprocess
from datetime import datetime
from pathlib import Path
from sklearn.calibration import CalibratedClassifierCV
from xgboost import XGBClassifier

def train_model(X_train, y_train, out_path="models/model.pkl"):
    scale = (len(y_train) - y_train.sum()) / y_train.sum()  # ~13
    base = XGBClassifier(
        n_estimators=200, max_depth=5, learning_rate=0.1,
        scale_pos_weight=scale, eval_metric="logloss",
        random_state=42, n_jobs=-1,
    )
    calibrated = CalibratedClassifierCV(base, method="isotonic", cv=3)
    calibrated.fit(X_train, y_train)

    Path(out_path).parent.mkdir(parents=True, exist_ok=True)
    sha = subprocess.check_output(["git", "rev-parse", "--short", "HEAD"]).decode().strip()
    joblib.dump({
        "model": calibrated,
        "features": list(X_train.columns),
        "trained_at": datetime.now().isoformat(),
        "git_sha": sha,
    }, out_path)
    return calibrated
Desafio 5.3 — Métricas + benchmark vs. baseline rule-based · arquivo: src/train.py + reports/model_metrics.json
📥 Entrada: modelo treinado + X_test, y_test + mart completo 📤 Saída: reports/model_metrics.json + print comparativo 🔗 Usado em: demo da Fase 5 + narrativa de entrevista
🎯 O que construir

Função evaluate_and_benchmark(model, X_test, y_test, df_test) que calcula 3 métricas para o modelo e para o baseline rule-based, e escreve o comparativo em JSON.

  • Métricas do modelo ML: roc_auc_score(y_test, y_proba), precision_at_k para k=10%/20% (os N% com maior pred_prob e a precision lá), log_loss(y_test, y_proba)
  • Métricas do baseline rule-based: tratar risk_tier == 'HIGH' como predição positiva. AUC não é bem-definido (saída binária), mas dá pra calcular AUC tratando a ordem LOW < MEDIUM < HIGH como score ordinal. Precision@k vira precision direta nos 13% de HIGH.
  • JSON de saída: {"model": {...}, "baseline_rule": {...}, "delta": {auc: ..., ...}, "n_test": ..., "pos_rate_test": ..., "trained_at": ...}
Por que precision@k e não só accuracy? Dataset com 6.8% de positivos: modelo que sempre prediz "NÃO default" acerta 93.2% das vezes. Accuracy é inútil. Precision@k=20% responde "dos 20% que o modelo MAIS suspeita, quantos realmente inadimplem?" — métrica de produto (se você só tem orçamento pra negar 20%, quantos verdadeiros positivos você pega?). É como times de risco pensam.
✅ Como testar

(a) JSON tem chaves model.auc, baseline_rule.auc, delta.auc — o último deve ser positivo (XGBoost > rule). (b) model.precision_at_20 deve ser > pos_rate_test (modelo melhor que random). (c) Print no console formato "AUC: XGB 0.862 vs rule 0.743 (+0.119)".

🎯 Expectativa de resultado

AUC do XGBoost geralmente fica em 0.82-0.88 nesse dataset (competition winners chegaram a ~0.87). AUC do baseline rule-based ordinal fica em 0.70-0.76. Delta esperado: +0.10 a +0.15. Precision@k=20% típico do XGB: 0.18-0.22 (2-3x acima do pos_rate).

📖 Referência — só abra após sua tentativa 
import json
from pathlib import Path
from sklearn.metrics import roc_auc_score, log_loss

TIER_ORDINAL = {"LOW": 0, "MEDIUM": 1, "HIGH": 2}

def precision_at_k(y_true, scores, k_pct):
    n = int(len(y_true) * k_pct)
    idx = scores.argsort()[::-1][:n]
    return y_true.iloc[idx].mean() if hasattr(y_true, "iloc") else y_true[idx].mean()

def evaluate_and_benchmark(model, X_test, y_test, df_test, out="reports/model_metrics.json"):
    y_proba = model.predict_proba(X_test)[:, 1]
    rule_score = df_test.loc[X_test.index, "risk_tier"].map(TIER_ORDINAL).values

    result = {
        "n_test": len(y_test),
        "pos_rate_test": float(y_test.mean()),
        "model": {
            "auc": float(roc_auc_score(y_test, y_proba)),
            "log_loss": float(log_loss(y_test, y_proba)),
            "precision_at_10": float(precision_at_k(y_test, y_proba, 0.10)),
            "precision_at_20": float(precision_at_k(y_test, y_proba, 0.20)),
        },
        "baseline_rule": {
            "auc": float(roc_auc_score(y_test, rule_score)),
            "precision_at_tier_high": float(y_test[df_test.loc[X_test.index, "risk_tier"] == "HIGH"].mean()),
        },
    }
    result["delta"] = {
        "auc": result["model"]["auc"] - result["baseline_rule"]["auc"],
    }
    Path(out).parent.mkdir(exist_ok=True)
    Path(out).write_text(json.dumps(result, indent=2, ensure_ascii=False))
    print(f'AUC: XGB {result["model"]["auc"]:.3f} vs rule {result["baseline_rule"]["auc"]:.3f} (+{result["delta"]["auc"]:.3f})')
    return result
Desafio 5.4 — Versionamento do modelo no GCS · ferramenta: gcloud storage
📥 Entrada: models/model.pkl + SHA do commit atual 📤 Saída: gs://credit-pipeline-demo-models/model-<sha>.pkl + model-latest.pkl 🔗 Consumido por: API carrega no startup (Bônus F5)
🎯 O que construir

Bucket GCS dedicado pra artefatos de modelo (separado do bucket do mart) + convenção de nome que versiona por SHA do commit. model-latest.pkl é um "alias" pra versão ativa.

  • Criar bucket: gcloud storage buckets create gs://credit-pipeline-demo-models --location=us-central1 --uniform-bucket-level-access --default-storage-class=STANDARD
  • Upload versionado: SHA=$(git rev-parse --short HEAD); gcloud storage cp models/model.pkl gs://credit-pipeline-demo-models/model-$SHA.pkl
  • Atualizar "latest": gcloud storage cp gs://credit-pipeline-demo-models/model-$SHA.pkl gs://credit-pipeline-demo-models/model-latest.pkl — é uma cópia explícita, não symlink; cloud storage não tem symlink
  • Grant read pra SA do Cloud Run: gcloud storage buckets add-iam-policy-binding gs://credit-pipeline-demo-models --member=serviceAccount:credit-pipeline-sa@... --role=roles/storage.objectViewer
Por que versionar por SHA e não por timestamp? Com SHA você consegue rastrear qual código treinou este modelo — basta git checkout do SHA pra reproduzir bit-a-bit. Timestamp perde essa conexão (se você treinou duas vezes no mesmo dia sem commit, qual data pegou?). Regra: artefatos de produção têm hash rastreável do código-fonte, não do wall clock.
✅ Como testar

(a) gcloud storage ls gs://credit-pipeline-demo-models/ mostra os dois arquivos (versionado e latest). (b) joblib.load do arquivo baixado funciona. (c) SA do Cloud Run consegue baixar (testar com gcloud auth application-default login com impersonation, ou direto no deploy do Bônus F5).

🎯 Expectativa de resultado

Bucket de modelos populado com ≥1 versão + latest. Custo: ~5-10 MB × R$0,10/GB/mês = desprezível. Pattern fica pronto pra retreinos futuros adicionarem novas versões sem apagar as antigas — rollback de modelo vira cp model-<sha-antigo>.pkl model-latest.pkl.
