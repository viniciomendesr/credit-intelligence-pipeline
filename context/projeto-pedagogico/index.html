Como usar este guia

Projeto em duas fases. Este guia serve aos dois momentos do aprendizado:

  • Fase 1 — Construção: o código de referência (gabarito) está pronto neste repo, rodando em produção. Você percorre os desafios aqui lendo e experimentando o que já está implementado.
  • Fase 2 — Redesenvolvimento: você cria um repositório separado e reimplementa cada desafio do zero, por 20-30 min antes de consultar o gabarito. É aqui que a teoria consolida — tropeçar é parte do processo.

Detalhes do setup do Redesenvolvimento (repo novo, estrutura mínima, fluxo por desafio) estão no README principal, seção Perfil 1.

4 desafios · SHAP · /explain-decision/ml · eval comparativo · demo runbook v2

🎯 Por que a progressão v1 → v2 (e não só um bônus)

Decisão de crédito no Brasil é caminho regulado (Res. 4.935/Bacen + CDC): o tomador tem direito a saber por que foi negado. Este projeto resolve isso em duas versões pedagogicamente distintas, porque os conceitos por trás — engenharia de AI vs. modelagem ML — são diferentes e merecem separação.

v1 (Bônus Fase 4) v2 (Bônus Fase 5) regra SQL da Fase 2 modelo XGBoost da Fase 5 │ │ ▼ ▼ risk_tier categórico pred_default_prob (0-1) │ │ ▼ ▼ extract_context extract_context_ml (desvio vs mediana) (SHAP values) │ │ ▼ ▼ LLM + guardrails LLM + guardrails │ │ ▼ ▼ /explain-decision/rule/:id /explain-decision/ml/:id ambos reusam: FastAPI + Cloud Run + cache por chave + eval programático

O que muda de v1 pra v2: só o input do extrator determinístico (heurística de desvio → SHAP). Todo o resto — endpoint, cache, eval com 3 checks, runbook, guardrails contra alucinação — fica idêntico. Na entrevista, isso vira: "resolvi o mesmo problema duas vezes porque queria sentir a diferença entre feature importance heurística e principiada."

Três casos de uso concretos que justificam o esforço:

  • Tela de negativa inline no SaaS parceiro — imobiliária, edtech ou healthtech renderiza "Sua dívida atual representa 78% da renda (mediana da carteira: 35%). Você pode contestar enviando comprovante adicional." em vez do genérico "negado por política interna". Reduz ticket de suporte e cumpre regulação.
  • Fila de revisão do analista de risco — decisões borderline caem numa fila; o analista abre, o dashboard bate em /explain-decision/ml/{id}, vê os 3 features com maior SHAP + narrativa técnica. Override em segundos em vez de dump de features.
  • Resposta a contestação / trilha de auditoria — se escalar pra jurídico ou Bacen, os key_factors estruturados servem de evidência auditável de que a decisão não foi baseada em atributo sensível.
"O endpoint transforma um score opaco em produto regulado. A separação entre extrator determinístico e narrador LLM é o que faz o compliance aprovar. Fiz v1 em cima da regra SQL pra sentir a ergonomia do padrão, depois refiz v2 em cima do modelo ML — e o eval confirma que ambas as versões atingem pass_rate >= 0.95. Tenho URL ao vivo com os dois endpoints."

Pitch resumido (qualquer audiência)

"Construí um pipeline de crédito de ponta a ponta: ingestão de CSV + API simulada de bureau, validação, transformação em camadas staging e mart com DuckDB, orquestração Prefect, logging estruturado, drift detection, deploy no Cloud Run, CI/CD com GitHub Actions. Em cima disso, treinei um XGBoost pra prever default e comparei contra o baseline rule-based (AUC do modelo vs. AUC da regra SQL). O diferencial final é explicabilidade em duas versões: /explain-decision/rule/{id} (v1, LLM narra a regra SQL) e /explain-decision/ml/{id} (v2, LLM narra o modelo com SHAP). Ambos os endpoints têm guardrails contra alucinação + eval programático que gate de CI rejeita PR se pass rate cair de 95%. Tenho URL ao vivo com os dois endpoints."
