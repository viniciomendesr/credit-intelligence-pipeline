✨ Bônus Fase 5 · Explicabilidade via LLM (v2 — modelo ML)
"O v1 narrava a regra SQL. Agora que existe um modelo treinado com AUC > da regra (Fase 5), o LLM pode narrar a probabilidade real de default com feature importance principiada (SHAP) em vez de heurística de desvio-vs-mediana. Como isso muda a narrativa e a eval?"

🧠 Por que refazer o explicável em cima do modelo (v2)

O Bônus Fase 4 entregou o padrão: extrator determinístico + narrador LLM com guardrails. Mas o extrator usava uma heurística — "top 3 features com maior desvio vs. mediana da carteira". Funciona pra didática, não resiste a revisão técnica: a mediana da carteira não é uma referência causal do risco. O modelo ML da Fase 5 tem feature importance principiada via SHAP values — cada feature recebe um "crédito" quantificado pela contribuição real à pred_default_prob daquele tomador específico.

"A v2 troca um único componente — a função que escolhe quais fatores mostrar — de heurística pra SHAP. Mesmo endpoint pattern, mesmo cache, mesmo prompt com guardrails, mesmo eval. O que ganho: explicações defensáveis em revisão de compliance, narrativas que citam probabilidade calibrada (34%) em vez de tier (MEDIUM), e a comparação side-by-side v1 vs v2 vira material de entrevista."

Entrega: src/decision_explainer_ml.py + endpoint GET /explain-decision/ml/{applicant_id} + script scripts/eval_explainer_ml.py espelhando os checks do v1 + runbook demo-bonus-fase5.md.

Pré-requisito: Fase 5 completa (models/model.pkl no GCS). Lib adicional: shap (pip install shap).

O que muda e o que não muda de v1 pra v2: o endpoint pattern é idêntico (cache por chave, 404, 503, shape de resposta). O prompt com guardrails é quase idêntico (só troca "valores do extrator" por "SHAP values do modelo"). O eval com 3 checks é template idêntico. O que muda de verdade é uma função: extract_context_ml usa SHAP, extract_context usava desvio-vs-mediana. Essa é a beleza da arquitetura — a separação extrator/narrador permite trocar só o extrator.
Desafio B5.1 — Extrator de reason-codes via SHAP + narrador LLM · arquivo: src/decision_explainer_ml.py
📥 Entrada: applicant_id + modelo carregado + mart 📤 Saída: dict com pred_default_prob, decision, key_factors (top-3 por |SHAP|), narrative, _usage 🔗 Chamado por: endpoint /explain-decision/ml do B5.2
🎯 O que construir

Duas funções em src/decision_explainer_ml.py:

  • extract_context_ml(applicant_id, mart, model_bundle): calcula pred_default_prob via model.predict_proba, usa shap.TreeExplainer pra extrair SHAP values da linha específica, seleciona as 3 features com maior |SHAP|, mapeia decisão a partir de thresholds (prob < 0.3 → APROVADO, 0.3-0.6 → APROVADO_COM_LIMITE, > 0.6 → NEGADO). Retorna dict determinístico.
  • explain_decision_ml(applicant_id, ...): chama extract_context_ml, monta prompt com os SHAP factors (incluindo sinal: positivo aumenta risco, negativo reduz), chama LLM com guardrails iguais ao v1, devolve dict com narrative, generated_at, _usage.
Por que SHAP e não só feature importance global? Feature importance global (model.feature_importances_) diz "renda é a feature mais importante no geral". SHAP diz "para este tomador específico, renda puxou a probabilidade em +0.12". A diferença: global é sobre o modelo; SHAP é sobre esta decisão. Pra explicar "por que VOCÊ foi negado", você precisa do local. É o que o Bacen quer e o que o SaaS parceiro renderiza no cliente final.
💡 Dica

shap.TreeExplainer(base_model) é ~100x mais rápido que KernelExplainer porque tem fast-path pra tree-based. Cuidado: CalibratedClassifierCV envelopa o XGB — pra SHAP você pega model.calibrated_classifiers_[0].estimator (o estimator base). SHAP da linha retorna array; concatenar com nomes de features via dict(zip(features, shap_values[0])).

🔧 Critério

(a) extract_context_ml é determinístico — mesmo id, mesma chamada, mesmos key_factors. (b) sum dos SHAP values + expected_value ≈ score pre-calibração do XGBoost (log-odds) — propriedade matemática que confirma que você usou SHAP corretamente. (c) A narrative cita pelo menos um número de key_factors (mesmo teste anti-alucinação do v1).

import os
from datetime import datetime
import joblib
import numpy as np
import pandas as pd
import shap
from anthropic import Anthropic

_CLIENT = None
_BUNDLE = None
_EXPLAINER = None

def _get_bundle(path="models/model.pkl"):
    global _BUNDLE, _EXPLAINER
    if _BUNDLE is None:
        _BUNDLE = joblib.load(path)
        base = _BUNDLE["model"].calibrated_classifiers_[0].estimator
        _EXPLAINER = shap.TreeExplainer(base)
    return _BUNDLE, _EXPLAINER

def _decision_from_prob(p: float) -> str:
    if p < 0.30: return "APROVADO"
    if p < 0.60: return "APROVADO_COM_LIMITE"
    return "NEGADO"

def extract_context_ml(applicant_id: int, mart: pd.DataFrame) -> dict:
    bundle, explainer = _get_bundle()
    model, features = bundle["model"], bundle["features"]
    row_df = mart[mart["applicant_id"] == applicant_id]
    if row_df.empty:
        raise ValueError(f"applicant_id {applicant_id} não encontrado no mart")
    X = row_df[features]
    prob = float(model.predict_proba(X)[0, 1])
    shap_vals = explainer.shap_values(X)[0]

    factors = [
        {"feature": f, "value": float(X.iloc[0][f]),
         "shap_value": float(shap_vals[i]),
         "direction": "aumenta_risco" if shap_vals[i] > 0 else "reduz_risco"}
        for i, f in enumerate(features)
    ]
    factors.sort(key=lambda d: abs(d["shap_value"]), reverse=True)
    return {
        "applicant_id": int(applicant_id),
        "pred_default_prob": round(prob, 4),
        "decision": _decision_from_prob(prob),
        "key_factors": factors[:3],
    }

def _build_prompt(ctx: dict) -> str:
    fs = "\n".join(
        f"- {f['feature']}={f['value']:.3f}  (SHAP={f['shap_value']:+.3f}, {f['direction']})"
        for f in ctx["key_factors"]
    )
    pct = ctx["pred_default_prob"] * 100
    return f"""Você é um analista de crédito de um motor de crédito B2B. Probabilidade estimada de inadimplência: {pct:.1f}%. Decisão: {ctx['decision']}.

Fatores com maior contribuição individual (SHAP values):
{fs}

Escreva explicação em português, máximo 3 frases, pro próprio tomador.

RESTRIÇÕES OBRIGATÓRIAS:
- Cite APENAS os valores/percentuais listados acima. Não invente.
- Não mencione raça, gênero, endereço, CEP ou atributo sensível.
- Não prometa reversão. Seja factual."""

def explain_decision_ml(applicant_id: int, mart: pd.DataFrame,
                        model: str = "claude-haiku-4-5-20251001") -> dict:
    global _CLIENT
    if _CLIENT is None:
        _CLIENT = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
    ctx = extract_context_ml(applicant_id, mart)
    resp = _CLIENT.messages.create(
        model=model, max_tokens=300,
        messages=[{"role": "user", "content": _build_prompt(ctx)}],
    )
    return {
        **ctx,
        "narrative": resp.content[0].text,
        "generated_at": datetime.now().isoformat(),
        "_usage": {
            "model": resp.model,
            "input_tokens": resp.usage.input_tokens,
            "output_tokens": resp.usage.output_tokens,
        },
    }
Desafio B5.2 — Endpoint /explain-decision/ml/{id} com cache por chave · arquivo: api/main.py
📥 Entrada: GET /explain-decision/ml/{applicant_id} 📤 Saída: JSON com pred_default_prob, decision, key_factors (SHAP), narrative, cached 🔗 Irmão: /explain-decision/rule/{id} (Bônus F4) — mesmo shape, extrator diferente
🎯 O que construir

Adicionar em api/main.py: carregamento do modelo no startup (mesmo padrão do mart) + endpoint /explain-decision/ml/{applicant_id} com cache por chave separado do cache do v1.

  • Load do modelo no startup: função _load_model() que baixa model-latest.pkl do bucket MODEL_BUCKET/MODEL_OBJECT (ADC), fallback pra models/model.pkl local em dev
  • Cache separado: _explanation_cache_ml: dict[int, dict] — não mistura com o cache do v1 porque a narrativa é diferente
  • Erros: 404 em id ausente, 503 em ANTHROPIC_API_KEY missing, 503 também em modelo não carregado (bucket indisponível)
Por que cache separado v1/v2? Mesmo applicant_id, dois endpoints, duas respostas diferentes. Compartilhar cache sobrescreveria uma com a outra. Duas chaves em um dict seria possível ({('rule', 42): ..., ('ml', 42): ...}) mas pra clareza pedagógica é mais honesto ter dois dicts separados e óbvios.
💡 Dica

A variável do modelo fica em escala de módulo (carregada uma vez por instância do Cloud Run). Se quiser hot-reload sem redeploy (trocar model-latest.pkl no GCS), adicione endpoint POST /reload-model — é uma variação. Pra produção de escala, usar Model Registry (Vertex AI), mas aqui é overkill.

🔧 Critério

(a) 3 curls sequenciais em 3 ids retornam cada um com cached: false; refazer retorna cached: true. (b) Comparar /explain-decision/rule/42 e /explain-decision/ml/42 — ambos funcionam, respostas têm estrutura similar mas números diferentes. (c) Testar com MODEL_BUCKET desconfigurado — fallback pra local deve funcionar em dev.

Desafio B5.3 — Eval do explicador ML em lote · arquivo: scripts/eval_explainer_ml.py
📥 Entrada: mart + modelo + 21 ids amostrados por faixa de pred_default_prob 📤 Saída: reports/eval_explainer_ml_<ts>.json com pass rates + custo + latência 🔗 Gate de CI: exit 1 se pass_rate_overall < 0.95
🎯 O que construir

Copia de eval_explainer_rule.py com 3 adaptações:

  • Amostragem: estratificar por faixa de pred_default_prob (baixo [0, 0.3), médio [0.3, 0.6), alto [0.6, 1]) em vez de risk_tier, porque agora o modelo é que decide.
  • Check aligned: se decision == NEGADO, pelo menos um fator no top-3 deve ter shap_value > 0 (empurrou prob pra cima) — mesma lógica do v1 (direction == alto_aumenta_risco), adaptada.
  • Pool de valid_values no grounded: incluir pred_default_prob (e *100), value de cada feature, shap_value, não mais "mediana da carteira" (não aplicável aqui).
Por que o template do v1 serve? A forma da resposta é a mesma: decisão + top-3 fatores + narrativa PT-BR. Os checks (grounded, aligned, forbidden) são propriedades de qualquer narrativa que cita fatos estruturados. Só os detalhes do pool de valores mudam. Se você teve que reescrever do zero, provavelmente abstraiu errado.
🔧 Critério

Rodar python scripts/eval_explainer_ml.py --n 21 --seed 42 gera reports/eval_explainer_ml_<ts>.json. Em run limpa: pass_rate_overall >= 0.95 e exit 0. Comparação side-by-side: abrir um eval_rule + um eval_ml e bater os campos pass_rate_* lado a lado — se a taxa do ML ficar muito abaixo, é sinal de prompt mal adaptado.

def check_aligned(result):
    if result["decision"] != "NEGADO":
        return True, ""
    has_driver = any(f["shap_value"] > 0 for f in result["key_factors"])
    return (True, "") if has_driver else (False, "NEGADO sem SHAP positivo no top-3")

def _expand_valid_values(ctx):
    out = [ctx["pred_default_prob"], ctx["pred_default_prob"] * 100]
    for f in ctx["key_factors"]:
        out.append(f["value"])
        out.append(f["shap_value"])
        if 0 < abs(f["value"]) <= 1:
            out.append(f["value"] * 100)
    return out
Desafio B5.4 — Demo ao vivo do explicador ML · arquivos: scripts/demo_bonus5_warmup.sh + demo-bonus-fase5.md
📥 Entrada: URL pública + 3 applicant_ids pré-validados (mesmos do v1 pra comparação) 📤 Saída: warm-up que pré-aquece v1 e v2 + runbook com beats de comparação side-by-side 🔗 Complementa: demo-bonus-fase4.md — o momento wow é a comparação, não cada um sozinho
🎯 O que construir

Dois artefatos que permitem mostrar v1 e v2 lado a lado no mesmo tomador:

  • scripts/demo_bonus5_warmup.sh: roda /health, dispara /explain-decision/rule/$id E /explain-decision/ml/$id pros mesmos 3 ids pré-validados. Abre Swagger, GitHub Actions, e os últimos eval_explainer_rule_*.json + eval_explainer_ml_*.json no editor.
  • demo-bonus-fase5.md: runbook com 3 beats — (1) v1 do tomador 37167, (2) v2 do mesmo id ao lado, (3) ponto alto abrindo os dois eval reports + reports/model_metrics.json da Fase 5 (o AUC +0.12).
Por que o demo definitivo é a comparação, não cada um isolado? Qualquer candidato mostra um endpoint de explicação. O diferencial é mostrar a mesma decisão explicada por duas lentes diferentes: "aqui é a regra SQL contando por que foi negado; aqui é o modelo ML contando a mesma coisa com probabilidade calibrada e SHAP. Reparem como a história muda." Em 90 segundos, você demonstra engenharia (mesmo padrão reusado), modelagem (treino + benchmark), e AI em produção (guardrails + eval).
💡 Dica

No Swagger, deixar dois endpoints expandidos ao mesmo tempo ajuda visualmente — o entrevistador enxerga lado a lado. Se preferir terminal, diff <(curl .../rule/$id | jq) <(curl .../ml/$id | jq) destaca o que mudou.

🔧 Critério

(a) O warm-up pré-aquece 6 combinações (3 ids × 2 endpoints) em <15s. (b) Cronometrando o runbook em voz alta, o demo v1+v2+eval+metrics cabe em 5 minutos. (c) Na parte de comparação, a narrativa muda de forma perceptível — se não, um dos dois prompts está mal adaptado.

#!/usr/bin/env bash
# scripts/demo_bonus5_warmup.sh
set -euo pipefail
URL="${1:-https://credit-api-10681834413.us-central1.run.app}"

ID_APROVADO="${ID_APROVADO:-37167}"
ID_LIMITE="${ID_LIMITE:-50821}"
ID_NEGADO="${ID_NEGADO:-23380}"

echo "▶ Warm-up v1 + v2: $URL"
for pair in "APROVADO:$ID_APROVADO" "LIMITE:$ID_LIMITE" "NEGADO:$ID_NEGADO"; do
  label="${pair%%:*}"; id="${pair##*:}"
  echo "▶ v1 rule  ($label, $id)"; curl -s "$URL/explain-decision/rule/$id" | jq '{decision, narrative: .narrative[:80]}'
  echo "▶ v2 ml    ($label, $id)"; curl -s "$URL/explain-decision/ml/$id"   | jq '{decision, pred_default_prob, narrative: .narrative[:80]}'
done

LATEST_RULE=$(ls -t reports/eval_explainer_rule_*.json 2>/dev/null | head -1 || true)
LATEST_ML=$(ls -t reports/eval_explainer_ml_*.json 2>/dev/null | head -1 || true)
METRICS=reports/model_metrics.json
if command -v open >/dev/null; then
  open "$URL/docs"
  [ -n "$LATEST_RULE" ] && command -v code >/dev/null && code "$LATEST_RULE" "$LATEST_ML" "$METRICS"
fi
echo "✅ Pronto. Siga o demo-bonus-fase5.md."
