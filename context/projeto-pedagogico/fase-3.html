Por que Prefect e não rodar os scripts na sequência? Um script sequencial falha silenciosamente — você não sabe em qual etapa foi. O Prefect dá visibilidade: cada task tem status, log e tempo de execução. Se a task "Fetch API" falhar, o flow para ali, loga o erro, e você sabe exatamente onde investigar. É a diferença entre um processo operacional e um pipeline monitorável.
Desafio 3.1 — Orquestrar com Prefect · arquivo: pipeline/flow.py
📥 Entrada: funções de src/ingestion.py + src/transform.py + tests/ 📤 Saída: flow executável com todas as etapas em sequência 🔗 Base para: logging e monitoramento da Fase 4
🎯 O que construir

Projete em pipeline/flow.py um orquestrador Prefect com 7 tasks em ordem fixa. Você não reescreve lógica — importa as funções das Fases 1 e 2 e as decora com @task.

  • Ordem obrigatória: ingest_csv → validate → fetch_api → merge → run_staging → run_mart → run_tests — cada função importada ganha @task(name="Nome Legível")
  • @flow: @flow(name="Credit Intelligence Pipeline") encadeia as tasks passando retornos como argumentos — df = ingest_csv() alimenta validate(df) diretamente
  • run_tests: subprocess.run(['pytest', 'tests/', '-v'], capture_output=True, text=True) — lança Exception se returncode != 0; o flow para no ponto exato da falha
O que você está fazendo aqui: as funções que você escreveu nas Fases 1 e 2 (validate_dataframe, merge_sources, run_model) são importadas e decoradas com @task. Você não reescreve a lógica — só orquestra. Esse é o princípio de separação de responsabilidades: a lógica fica nos módulos, o orquestrador só define a ordem.

Desafio 3.2 — API FastAPI · arquivo: api/main.py
📥 Entrada: data/marts/mart_credit_features.parquet (gerado pelo flow) 📤 Saída: endpoints REST em localhost:8000 🔗 Containerizada por: Dockerfile do 3.3
🎯 O que construir

Projete em api/main.py uma FastAPI com dois endpoints. O Parquet é carregado no nível do módulo — uma vez no startup, não a cada request.

  • Carregamento: df = pd.read_parquet('data/marts/mart_credit_features.parquet') no topo do arquivo, fora de qualquer função — ler 150k linhas por request adicionaria ~200ms de latência desnecessária
  • GET /health: retorna {"status": "ok", "timestamp": datetime.now().isoformat(), "records": len(df)}
  • GET /risk-summary: retorna default_rate_pct (mean * 100, round 2), risk_tier_distribution_pct (value_counts(normalize=True).mul(100).round(2).to_dict()), e median_income_by_tier (groupby('risk_tier')['monthly_income'].median().round(2).to_dict())
  • Validação: default_rate_pct deve bater com o benchmark do DuckDB 2.1 (margem de ±0.1% por arredondamento) — se divergir mais, há um bug na lógica ou nos dados
Carrega o Parquet uma vez, não a cada request: o df = pd.read_parquet(...) fica no nível do módulo — executado quando a API sobe, não a cada chamada. Ler 150k linhas a cada request adicionaria ~200ms de latência desnecessária. Variável global é o padrão aqui.

Desafio 3.3 — Dockerizar · arquivo: Dockerfile
📥 Entrada: api/, src/, data/marts/, requirements.txt 📤 Saída: imagem Docker com a API rodando na porta 8000 🔗 Publicada no: GCP Artifact Registry na Fase 4
🎯 O que construir

Projete um Dockerfile baseado em python:3.11-slim que empacota a API com os dados Parquet embutidos. A ordem das instruções é deliberada — maximiza o cache de layers do Docker.

  • Dependências primeiro: COPY requirements.txt . + RUN pip install --no-cache-dir -r requirements.txt antes de qualquer código — se só o código mudar, o Docker reutiliza o layer do pip sem reinstalar
  • Código e dados: COPY api/ ./api/, COPY src/ ./src/, COPY data/marts/ ./data/marts/ — sem o terceiro, o Parquet não existe no container e a API falha no startup
  • CMD: ["uvicorn", "api.main:app", "--host", "0.0.0.0", "--port", "8000"] — sem 0.0.0.0, o servidor escuta só em localhost dentro do container e o -p 8000:8000 não entrega tráfego
Por que incluir os dados na imagem? A API lê o Parquet no startup. Em produção você teria um volume ou storage bucket, mas para este projeto incluir o Parquet na imagem mantém tudo autocontido — você consegue rodar com um único docker run sem configuração adicional. O tradeoff é que a imagem fica maior.
📋 Guia passo a passo — abra só se travar
Passo 1 de 3 Definir base image e instalar dependências

Comece com FROM python:3.11-slim — slim é menor que a imagem completa pois não tem compiladores e ferramentas de build. Copie e instale dependências antes do código: COPY requirements.txt . então RUN pip install --no-cache-dir -r requirements.txt. O --host 0.0.0.0 é obrigatório em container — sem ele a API escuta só em localhost dentro do container e não recebe tráfego externo. Antes de rodar: teste localmente com docker build -t credit-api . && docker run -p 8000:8000 credit-api antes de avançar para o Cloud Run.
