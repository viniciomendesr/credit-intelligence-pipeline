Desafio 2.2 — Modelagem em camadas (Staging → Mart) · arquivos: models/*.sql
📥 Entrada: data/staging/credit_applications.parquet 📤 Saída: models/stg_credit_applications.sql + models/mart_credit_features.sql 🔗 Executados por: run_model() no 2.3
🎯 O que construir

Projete dois arquivos SQL em models/ com responsabilidades distintas: a staging limpa e renomeia, a mart deriva features para ML. A mart consome a staging por nome — não o Parquet diretamente.

  • Staging — limpeza: renomear para snake_case (SeriousDlqin2yrs AS defaulted, MonthlyIncome AS monthly_income, etc.), NOW() AS loaded_at para rastreabilidade, WHERE age BETWEEN 18 AND 100 AND monthly_income >= 0 — zero é válido, negativo não é
  • Staging — deduplicação: QUALIFY ROW_NUMBER() OVER (PARTITION BY age, monthly_income, debt_ratio ORDER BY 1) = 1 no final — sem subconsulta nem CTE; ORDER BY 1 apenas para satisfazer a sintaxe
  • Mart — features derivadas: total_late_payments = late_30_59_days + late_60_89_days + late_90_days; has_90day_default = CASE WHEN late_90_days > 0 THEN 1 ELSE 0 END
  • Mart — risk_tier: HIGH se revolving_utilization > 0.9 OR late_90_days > 2, MEDIUM se revolving_utilization > 0.5 OR late_30_59_days > 1, LOW nos demais — validar que nenhuma categoria supera 90% dos registros
Por que dois arquivos separados? A staging lê direto do Parquet da Fase 1. A mart lê da staging. Essa separação significa que se a lógica de risk_tier mudar, você mexe só na mart sem tocar na limpeza — e a staging pode ser reusada por outros marts no futuro. O run_model() do 2.3 cria a staging como view temporária antes de executar a mart.

SELECT
    ROW_NUMBER() OVER ()                     AS applicant_id,
    SeriousDlqin2yrs                         AS defaulted,
    RevolvingUtilizationOfUnsecuredLines     AS revolving_utilization,
    age, DebtRatio AS debt_ratio,
    MonthlyIncome AS monthly_income,
    NumberOfOpenCreditLinesAndLoans          AS open_credit_lines,
    NumberOfTimes90DaysLate                  AS late_90_days,
    NumberOfTime30_59DaysPastDueNotWorse     AS late_30_59_days,
    NumberOfTime60_89DaysPastDueNotWorse     AS late_60_89_days,
    NumberOfDependents AS dependents, income_missing,
    NOW() AS loaded_at
FROM 'data/staging/credit_applications.parquet'
WHERE age BETWEEN 18 AND 100 AND monthly_income >= 0
QUALIFY ROW_NUMBER() OVER (PARTITION BY age, monthly_income, debt_ratio ORDER BY 1) = 1
-- models/mart_credit_features.sql
-- Nota: depende de stg_credit_applications (criada como view pelo run_model)
SELECT
    applicant_id, defaulted, age, monthly_income, revolving_utilization,
    debt_ratio, open_credit_lines, dependents, income_missing,
    (late_30_59_days + late_60_89_days + late_90_days) AS total_late_payments,
    CASE WHEN late_90_days > 0 THEN 1 ELSE 0 END AS has_90day_default,
    CASE
        WHEN revolving_utilization > 0.9 OR late_90_days > 2 THEN 'HIGH'
        WHEN revolving_utilization > 0.5 OR late_30_59_days > 1 THEN 'MEDIUM'
        ELSE 'LOW'
    END AS risk_tier,
    loaded_at
FROM stg_credit_applications
Desafio 2.3 — Executar modelos SQL via Python · arquivo: src/transform.py
📥 Entrada: models/*.sql + Parquet de staging 📤 Saída: data/marts/mart_credit_features.parquet 🔗 Importada por: flow.py do Prefect na Fase 3
🎯 O que construir

Projete run_model(model_name, output_path) em src/transform.py — executa um arquivo SQL via DuckDB e materializa o resultado como Parquet. A função é idempotente: rodar duas vezes não gera erro.

  • Diretório de saída: Path(output_path).mkdir(parents=True, exist_ok=True) — sem isso, to_parquet() falha com FileNotFoundError
  • Staging como view: CREATE OR REPLACE VIEW stg_credit_applications AS {stg_sql} — sempre, antes de qualquer modelo, porque a mart referencia esse nome
  • Execução e persistência: con.execute(sql).df() e .to_parquet(out, index=False) — index=False evita coluna extra que quebraria o schema da API
Como a mart enxerga a staging? O run_model() sempre cria a staging como view temporária antes de executar qualquer modelo — assim a mart pode usar FROM stg_credit_applications normalmente. É exatamente como o dbt orquestra dependências entre modelos.
📋 Guia passo a passo — abra só se travar
Passo 1 de 3 Ler o arquivo SQL como texto

Leia com Path(f'models/{model_name}.sql').read_text() — sem parser, sem template engine, só texto puro que vai para o DuckDB. Antes de escrever: o que acontece se o arquivo não existir? Python lança FileNotFoundError — e o usuário vê exatamente em qual modelo ocorreu o problema.

Passo 2 de 3 Criar staging como view antes de rodar qualquer modelo

Execute con.execute(f"CREATE OR REPLACE VIEW stg_credit_applications AS {stg_sql}") — sempre, independente do modelo solicitado. Assim a mart pode usar FROM stg_credit_applications normalmente. Antes de rodar: por que CREATE OR REPLACE e não CREATE? O que acontece se rodar run_model() duas vezes?

Passo 3 de 3 Executar e salvar como Parquet

Execute com con.execute(sql).df() para ter um DataFrame pandas, depois salve com result.to_parquet(out, index=False). O index=False evita uma coluna extra de índice que quebraria a leitura do Parquet na FastAPI. Antes de rodar: o que é diferente no arquivo se você usar index=True?

import duckdb
from pathlib import Path

def run_model(model_name, output_path):
    con = duckdb.connect()
    Path(output_path).mkdir(parents=True, exist_ok=True)
    # Sempre cria staging como view (mart depende dela)
    stg = Path('models/stg_credit_applications.sql').read_text()
    con.execute(f"CREATE OR REPLACE VIEW stg_credit_applications AS {stg}")
    # Executa o modelo solicitado
    sql = Path(f'models/{model_name}.sql').read_text()
    result = con.execute(sql).df()
    out = Path(output_path) / f"{model_name}.parquet"
    result.to_parquet(out, index=False)
    print(f"'{model_name}': {len(result)} registros → {out}")
Desafio 2.4 — Testes de qualidade com pytest · arquivo: tests/test_pipeline.py
📥 Entrada: data/marts/mart_credit_features.parquet 📤 Saída: 3 testes — verde significa dado válido para produção 🔗 Executado por: Prefect (Fase 3) + GitHub Actions (Fase 4)
🎯 O que construir

Projete em tests/test_pipeline.py uma suite pytest com 3 testes de contrato de dados. A @pytest.fixture carrega o Parquet e injeta o DataFrame em cada teste pelo nome do parâmetro — não uma variável global.

  • Teste 1 — nulos: itere ['applicant_id', 'defaulted', 'risk_tier', 'total_late_payments'] com assert mart[col].isnull().sum() == 0, f"Nulos em {col}" — a mensagem identifica a coluna exata
  • Teste 2 — valores de risk_tier: set(mart['risk_tier'].unique()).issubset({'LOW', 'MEDIUM', 'HIGH'}) — .issubset() captura typos como 'high' e None que um == simples deixaria passar
  • Teste 3 — total_late_payments: (mart['total_late_payments'] >= 0).all() — o .all() é obrigatório; sem ele, a asserção nunca falha independente dos dados
Por que testar o Parquet e não o código? Você testa o dado porque é ele que vai alimentar a API e os modelos. Um teste que passa não garante que o código é perfeito — garante que o resultado atende ao contrato de qualidade. Na Fase 3, o Prefect vai rodar pytest como última task: se falhar, o pipeline para. Na Fase 4, o GitHub Actions faz o mesmo antes de qualquer deploy.
