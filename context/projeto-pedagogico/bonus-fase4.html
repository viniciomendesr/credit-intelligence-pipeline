✨ Bônus Fase 4 · Explicabilidade via LLM (v1 — regra SQL)
"E se a API não só devolvesse a decisão de crédito — mas também explicasse, em PT-BR, por que aquele tomador específico foi aprovado, limitado ou negado?"
v1: decisão vem da regra SQL da Fase 2. Na Fase 5 treina-se um modelo; no Bônus Fase 5 o LLM narra o modelo.

🧠 Por que isso muda o nível da conversa na entrevista

Motores de crédito B2B vendem a camada de inteligência que entrega "crédito em segundos" para SaaS verticais — e decisão de crédito no Brasil é caminho regulado: o tomador tem direito a saber por que foi negado. Este módulo constrói exatamente isso: um endpoint que recebe um application_id, busca os fatores determinantes no mart, e devolve uma narrativa em PT-BR gerada por LLM com guardrails explícitos contra alucinação. Não é dashboard narrado — é reason-code em produção, o mesmo tipo de coisa que vira ticket no board do time.

"O pipeline tem um endpoint de explicação de decisão: você passa o id do tomador, o motor extrai deterministicamente os 3 fatores de maior peso no score, e o LLM narra a decisão em PT-BR sem poder inventar número — só pode citar os valores que eu passei no contexto. Tenho URL ao vivo."

Entrega: src/decision_explainer_rule.py + endpoint GET /explain-decision/rule/{applicant_id} na API FastAPI retornando decisão + narrativa LLM com guardrails + script scripts/eval_explainer_rule.py que roda eval programático em lote e gera relatório de eficácia + runbook de demo ao vivo.

Pré-requisito: mart_credit_features.parquet gerado (Fase 2) com colunas risk_tier, defaulted, features de renda/dívida/inadimplência. ANTHROPIC_API_KEY ou OPENAI_API_KEY como variável de ambiente.

O que liga tudo: o mart_credit_features.parquet da Fase 2 é o input do extrator. A função determinística decide o que o LLM pode falar; o LLM decide como falar. Essa separação é o que torna reason-code LLM utilizável em produção — sem ela, o modelo inventa número e o time jurídico veta. O endpoint /explain-decision reusa a API da Fase 4, sem novo serviço.
Desafio B.1 — Extrator de reason-codes + narrador LLM · arquivo: src/decision_explainer.py
📥 Entrada: application_id (int) + mart parquet 📤 Saída: dict com decision, risk_tier, key_factors, narrative 🔗 Chamado por: endpoint /explain-decision do B.2
🎯 O que construir

Duas funções em src/decision_explainer.py:

  • extract_context(application_id, mart_path): lê o mart, localiza a linha, mapeia risk_tier → decisão (LOW→APROVADO, MEDIUM→APROVADO_COM_LIMITE, HIGH→NEGADO), e calcula os 3 fatores com maior desvio em relação à mediana da carteira (ratio absoluto). Retorna dict determinístico — sem LLM ainda.
  • explain_decision(application_id, mart_path): chama extract_context, monta o prompt com os fatores, chama o LLM com restrição explícita ("cite APENAS os valores acima, não invente número"), devolve dict com os campos do contexto + narrative + generated_at.
Por que separar extração e narração? Se o LLM decide sozinho quais fatores mencionar, ele seleciona o que parece plausível — e pode omitir uma feature crítica ou inventar uma que não existe. Com a extração determinística antes, você controla o que pode ser dito; o LLM só controla o tom e a estrutura da frase. Essa separação é o que permite passar revisão de compliance — é a mesma lógica que bancos usam no reason-code de negativa de crédito.
💡 Dica

Desvio vs mediana: ratio = (valor - mediana) / (abs(mediana) + 1e-9), ordena por abs(ratio) desc, pega top 3. Guardrails no prompt importam: "use apenas os valores listados", "não mencione raça, gênero, CEP", "máximo 3 frases". Se o mart tiver <10 linhas (teste unitário), a mediana degenera — retornar erro explícito.

🔧 Critério

Dois testes que separam lógica de LLM: (a) test_extract_context_deterministic — dado o mesmo application_id, extract_context retorna sempre os mesmos key_factors. (b) test_narrative_grounded — a narrative contém pelo menos um dos valores numéricos dos key_factors (regex na string). Esse segundo teste é o anti-alucinação mínimo.

import os
from datetime import datetime
from pathlib import Path
import pandas as pd
from anthropic import Anthropic

client = Anthropic(api_key=os.environ.get("ANTHROPIC_API_KEY"))

DECISION_BY_TIER = {
    "LOW":    "APROVADO",
    "MEDIUM": "APROVADO_COM_LIMITE",
    "HIGH":   "NEGADO",
}

# (feature_no_mart, label_humano, direção_no_risco)
FEATURES = [
    ("debt_ratio",       "Razão dívida/renda",           "alto_aumenta_risco"),
    ("monthly_income",   "Renda mensal declarada",       "alto_reduz_risco"),
    ("age",              "Idade",                        "neutro"),
    ("num_late_60_89",   "Atrasos recentes (60-89 dias)", "alto_aumenta_risco"),
    ("income_missing",   "Renda não declarada",          "alto_aumenta_risco"),
]

def extract_context(application_id: int,
                    mart_path: str = "data/marts/mart_credit_features.parquet") -> dict:
    df = pd.read_parquet(mart_path)
    if not (0 <= application_id < len(df)):
        raise ValueError(f"application_id {application_id} fora de range (0..{len(df)-1})")

    row = df.iloc[application_id]
    cols = [f[0] for f in FEATURES if f[0] in df.columns]
    medians = df[cols].median()

    factors = []
    for col, label, direction in FEATURES:
        if col not in df.columns:
            continue
        val = float(row[col])
        med = float(medians[col])
        ratio = (val - med) / (abs(med) + 1e-9)
        factors.append({
            "feature": col, "label": label,
            "value": round(val, 2), "median": round(med, 2),
            "deviation_ratio": round(ratio, 2), "direction": direction,
        })
    factors.sort(key=lambda f: abs(f["deviation_ratio"]), reverse=True)

    return {
        "application_id": application_id,
        "risk_tier": row["risk_tier"],
        "decision": DECISION_BY_TIER[row["risk_tier"]],
        "key_factors": factors[:3],
    }

def _build_prompt(ctx: dict) -> str:
    factors_text = "\n".join(
        f"- {f['label']}: {f['value']} (mediana da carteira: {f['median']})"
        for f in ctx["key_factors"]
    )
    return f"""Você é um analista de crédito de um motor de crédito B2B. Um tomador recebeu a decisão {ctx['decision']} (tier de risco {ctx['risk_tier']}).

Fatores extraídos pelo motor de risco:
{factors_text}

Escreva uma explicação em português, máximo 3 frases, dirigida ao próprio tomador.

RESTRIÇÕES OBRIGATÓRIAS:
- Cite APENAS os valores listados acima. Não invente números.
- Não mencione raça, gênero, endereço, CEP ou qualquer atributo sensível.
- Não prometa reversão da decisão. Seja factual, não motivacional."""

def explain_decision(application_id: int,
                     mart_path: str = "data/marts/mart_credit_features.parquet") -> dict:
    ctx = extract_context(application_id, mart_path)
    prompt = _build_prompt(ctx)
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=300,
        messages=[{"role": "user", "content": prompt}],
    )
    return {
        **ctx,
        "narrative": response.content[0].text,
        "generated_at": datetime.now().isoformat(),
        # Observabilidade do LLM — consumido pelo eval (B.3) e por dashboards de custo
        "_usage": {
            "model": response.model,
            "input_tokens":  response.usage.input_tokens,
            "output_tokens": response.usage.output_tokens,
        },
    }
Desafio B.2 — Endpoint /explain-decision com cache por chave · arquivo: api/main.py
📥 Entrada: GET /explain-decision/rule/{application_id} 📤 Saída: JSON com decision + key_factors + narrative + cached 🔗 Adicionado em: api/main.py já deployado no Cloud Run
🎯 O que construir

Adicione GET /explain-decision/rule/{application_id} em api/main.py que chama explain_decision() e retorna o resultado com cached: bool. Cache em dict Python indexado por application_id, TTL de 30 minutos. Se o id estiver fora de range, devolve HTTPException 404.

Por que cache por chave e não global? O endpoint /insights das versões anteriores deste bônus cachava uma única resposta global com TTL curto — fazia sentido porque a entrada era sempre o mesmo arquivo de métricas. Aqui a entrada varia por application_id: cachear globalmente significa que o tomador 7 receberia a explicação do tomador 3. Cache por chave é o padrão correto toda vez que a resposta depende do input. TTL de 30min reflete que a explicação de uma decisão específica não muda (mesmo mart, mesmo modelo, mesmo prompt = mesma saída) — mas você quer invalidar se o modelo de risco for retreinado.
💡 Dica

_explanation_cache: dict[int, dict] = {{}} na escala de módulo. Guarda _cached_at como campo interno; filtra antes de devolver. No Cloud Run com múltiplas instâncias cada réplica tem seu próprio cache — aceitável aqui (explicação é idempotente), mas a variação abaixo discute quando isso vira problema.

🔧 Critério

Três curl em sequência: /explain-decision/7 → cached: false; /explain-decision/7 de novo → cached: true e generated_at idêntico; /explain-decision/8 → cached: false (chave diferente). /explain-decision/999999 → 404 com detail explicativo. Após o deploy, a URL pública também responde — testa em https://<sua-url>/explain-decision/7.

# Adiciona em api/main.py junto com os endpoints existentes
from datetime import datetime, timedelta
from fastapi import HTTPException
import sys
sys.path.append('.')
from src.decision_explainer import explain_decision

_explanation_cache: dict[int, dict] = {}
EXPLANATION_TTL_MINUTES = 30

@app.get("/explain-decision/rule/{application_id}")
def explain(application_id: int):
    now = datetime.now()
    cached = _explanation_cache.get(application_id)
    if cached and (now - cached["_cached_at"]) < timedelta(minutes=EXPLANATION_TTL_MINUTES):
        return {**{k: v for k, v in cached.items() if k != "_cached_at"}, "cached": True}

    try:
        result = explain_decision(application_id)
    except ValueError as e:
        raise HTTPException(status_code=404, detail=str(e))

    _explanation_cache[application_id] = {**result, "_cached_at": now}
    return {**result, "cached": False}
Desafio B.3 — Eval do explicador em lote · arquivo: scripts/eval_explainer.py
📥 Entrada: mart parquet + N amostras (default 21, estratificadas por tier) 📤 Saída: reports/eval_explainer_<timestamp>.json com pass_rate por check, latência p95, custo total em USD, lista de violações 🔗 Gate de CI: sai com exit 1 se pass_rate_overall < 0.95
🎯 O que construir

Script que amostra N application_ids estratificados (proporção do tier na carteira: ~70% LOW, 16% MEDIUM, 14% HIGH — default 15/3/3), chama explain_decision em cada um, e aplica três checks programáticos:

  • Check 1 — Grounded-ness: todo número que aparece na narrative (extraído via regex r"\d+[.,]?\d*") precisa existir entre os value/median dos key_factors (com tolerância de arredondamento). Detecta alucinação numérica.
  • Check 2 — Alinhamento decisão↔fator: se decision == "NEGADO", pelo menos um fator com direction == "alto_aumenta_risco" deve estar no top-3. Detecta o caso em que o LLM narra uma coisa e o extrator apontou outra — sintoma de prompt mal construído.
  • Check 3 — Forbidden content: a narrative não pode conter nenhuma substring de uma lista fixa (["cep", "endereço", "raça", "gênero", "estado civil", "religião"], case-insensitive). Checagem de compliance mínima.

O relatório final agrega: pass_rate_grounded, pass_rate_aligned, pass_rate_forbidden, pass_rate_overall (AND dos três), latency_p95_ms, cost_usd_total, violations (lista de {application_id, check, detail}). Se pass_rate_overall < 0.95, exit 1 — gate de CI.

Por que eval automatizado, não revisão manual? O problema com LLM em produção não é que ele erra — é que erra de forma silenciosa e não-determinística. Uma mudança de prompt ou troca de modelo pode passar "vibe check" e introduzir alucinação em 5% dos casos sem ninguém perceber. Eval em lote com checks binários é a única forma de detectar regressão antes que um tomador contestado pelo Bacen descubra. Em time real, esse script roda em PR hook — se a taxa cai abaixo de 95%, o PR não faz merge.
💡 Dica

Latência por chamada: t0 = time.perf_counter() antes, delta depois; agrega com numpy.percentile(latencies, 95). Custo: conta tokens de entrada/saída do response.usage da SDK Anthropic e multiplica pelo preço tabelado do Haiku (custo de input + output). Estratificação: df.groupby('risk_tier').sample(n=...) com random_state fixo pra reprodutibilidade.

🔧 Critério

Rodar python scripts/eval_explainer.py --n 21 --seed 42 gera um arquivo em reports/ com os campos acima. Em uma run limpa com prompt com guardrails: pass_rate_overall >= 0.95 e exit 0. Pra validar o gate: remove a restrição "não invente números" do prompt em decision_explainer.py, roda o eval, confirma que pass_rate_grounded cai e exit code vira 1.

import argparse, json, re, sys, time
from datetime import datetime
from pathlib import Path
import numpy as np
import pandas as pd
from src.decision_explainer import explain_decision

# Preço Haiku 4.5 (ajustar conforme tabela vigente)
PRICE_IN_PER_MTOK  = 1.00   # USD / milhão tokens input
PRICE_OUT_PER_MTOK = 5.00   # USD / milhão tokens output

FORBIDDEN = ["cep", "endereço", "raça", "gênero", "estado civil", "religião"]
NUM_RE = re.compile(r"\d+[.,]?\d*")


def stratified_sample(mart_path: str, n: int, seed: int) -> list[int]:
    df = pd.read_parquet(mart_path)
    # Proporção aproximada da carteira
    props = {"LOW": 0.70, "MEDIUM": 0.16, "HIGH": 0.14}
    ids = []
    for tier, p in props.items():
        k = max(1, round(n * p))
        sample = df[df["risk_tier"] == tier].sample(n=k, random_state=seed)
        ids.extend(sample.index.tolist())
    return ids[:n]


def check_grounded(narrative: str, key_factors: list[dict]) -> tuple[bool, str]:
    valid_nums = set()
    for f in key_factors:
        valid_nums.add(f"{f['value']:.2f}")
        valid_nums.add(str(int(f['value'])) if f['value'].is_integer() else f"{f['value']:.2f}")
        valid_nums.add(f"{f['median']:.2f}")
    nums_in_text = NUM_RE.findall(narrative)
    for n in nums_in_text:
        n_norm = n.replace(",", ".")
        try:
            val = float(n_norm)
            if not any(abs(val - float(v.replace(",", "."))) < 0.01 for v in valid_nums):
                return False, f"número '{n}' não veio do contexto"
        except ValueError:
            continue
    return True, ""


def check_aligned(result: dict) -> tuple[bool, str]:
    if result["decision"] != "NEGADO":
        return True, ""
    has_risk_driver = any(
        f["direction"] == "alto_aumenta_risco" for f in result["key_factors"]
    )
    return (True, "") if has_risk_driver else (False, "NEGADO sem fator de risco no top-3")


def check_forbidden(narrative: str) -> tuple[bool, str]:
    low = narrative.lower()
    hit = next((w for w in FORBIDDEN if w in low), None)
    return (False, f"menciona '{hit}'") if hit else (True, "")


def run_eval(mart_path: str, n: int, seed: int) -> dict:
    ids = stratified_sample(mart_path, n, seed)
    results, latencies, violations = [], [], []
    cost_in = cost_out = 0
    pass_grounded = pass_aligned = pass_forbidden = 0
    pass_all = 0

    for app_id in ids:
        t0 = time.perf_counter()
        r = explain_decision(app_id, mart_path)
        latencies.append((time.perf_counter() - t0) * 1000)

        usage = r.get("_usage", {})
        cost_in  += usage.get("input_tokens", 0)
        cost_out += usage.get("output_tokens", 0)

        ok_g, msg_g = check_grounded(r["narrative"], r["key_factors"])
        ok_a, msg_a = check_aligned(r)
        ok_f, msg_f = check_forbidden(r["narrative"])

        pass_grounded  += ok_g
        pass_aligned   += ok_a
        pass_forbidden += ok_f
        pass_all       += (ok_g and ok_a and ok_f)

        for ok, check, msg in [(ok_g,"grounded",msg_g), (ok_a,"aligned",msg_a), (ok_f,"forbidden",msg_f)]:
            if not ok:
                violations.append({"application_id": app_id, "check": check, "detail": msg})

    total = len(ids)
    return {
        "n_samples": total,
        "seed": seed,
        "pass_rate_grounded":  round(pass_grounded / total, 4),
        "pass_rate_aligned":   round(pass_aligned / total, 4),
        "pass_rate_forbidden": round(pass_forbidden / total, 4),
        "pass_rate_overall":   round(pass_all / total, 4),
        "latency_p95_ms":      round(float(np.percentile(latencies, 95)), 2),
        "cost_usd_total":      round(cost_in * PRICE_IN_PER_MTOK / 1e6
                                    + cost_out * PRICE_OUT_PER_MTOK / 1e6, 6),
        "violations": violations,
        "generated_at": datetime.now().isoformat(),
    }


if __name__ == "__main__":
    ap = argparse.ArgumentParser()
    ap.add_argument("--mart", default="data/marts/mart_credit_features.parquet")
    ap.add_argument("--n", type=int, default=21)
    ap.add_argument("--seed", type=int, default=42)
    ap.add_argument("--threshold", type=float, default=0.95)
    args = ap.parse_args()

    report = run_eval(args.mart, args.n, args.seed)
    out = Path("reports") / f"eval_explainer_{datetime.now():%Y%m%d_%H%M%S}.json"
    out.parent.mkdir(exist_ok=True)
    out.write_text(json.dumps(report, indent=2, ensure_ascii=False))
    print(json.dumps({k: v for k, v in report.items() if k != "violations"}, indent=2))
    print(f"Relatório completo: {out}")

    if report["pass_rate_overall"] < args.threshold:
        print(f"❌ FAIL: pass_rate_overall {report['pass_rate_overall']} < threshold {args.threshold}")
        sys.exit(1)
    print(f"✅ PASS")
Desafio B.4 — Demo ao vivo do endpoint explicável · arquivos: scripts/demo_bonus_warmup.sh + demo-bonus.md
📥 Entrada: URL pública + 3 application_ids pré-validados (1 por tier) + último reports/eval_explainer_rule_*.json 📤 Saída: script de warm-up + runbook de 5 min focado no explicador + ponto alto com o eval report 🔗 Complementa: o runbook demo.md da Fase 4.5 (sistema sem LLM) — aqui entra o diferencial do LLM com guardrails
🎯 O que construir

Dois artefatos que transformam o /explain-decision em demo de 5 minutos que não falha — seja em validação pessoal pós-deploy, showcase para mentor/colaborador, portfólio ou entrevista técnica:

  • scripts/demo_bonus_warmup.sh: roda /health, dispara /explain-decision/rule/{id} para os 3 ids pré-validados (pré-aquece o LLM client + popula o cache por chave do B.2), e abre Swagger, GitHub Actions e o último eval report em reports/ no editor. Na segunda chamada ao vivo, a resposta volta em ~30ms com cached: true — quem estiver assistindo vê isso.
  • demo-bonus.md na raiz: runbook de 5 minutos com 3 beats específicos do LLM — (1) explicação de uma decisão APROVADA, (2) explicação de uma NEGADA (mais impactante — mostra os fatores de risco narrados em PT-BR), (3) ponto alto abrindo o eval_explainer_rule_*.json e falando a linha sobre "confiança em LLM não vem de boa fé, vem do relatório que roda no CI". Inclui respostas para 3 perguntas LLM-específicas.
Por que o eval report é o ponto alto da demo? Plugar uma API LLM num endpoint e mostrar "olha, gerei texto" é fácil. O que separa é você abrir um JSON que diz pass_rate_grounded: 1.0, cost_usd_total: 0.0043, latency_p95_ms: 1847 em 21 amostras estratificadas e dizer "esse relatório roda como gate de CI. Se grounded-ness cair de 95%, o PR não passa". Em 30 segundos você demonstra três coisas de uma vez: que sabe o que é alucinação, que sabe o que é custo, e que sabe o que é regressão de prompt. Nenhum destes vem da API em si — vem do B.3 que você construiu.
💡 Dica

Pra escolher os 3 ids pré-validados, roda o eval do B.3 uma vez com --n 30, varre o JSON e escolhe 1 de cada tier cuja narrativa você revisou e aprovou manualmente — guarda esses 3 ids como constantes. Nunca digite id "qualquer" ao vivo: tier HIGH com dado estranho pode gerar narrativa que soa mal no momento da apresentação. Pro ponto alto, abre o JSON em editor com JSON fold colapsado — quando expande o campo violations: [] vazio, o ponto fica visível sozinho.

🔧 Critério

(a) bash scripts/demo_bonus_warmup.sh <URL> termina em <10s, as 3 explicações são JSON válido com narrative preenchido, o eval report mais recente abre no editor. (b) Imediatamente após o warm-up, refazendo curl em qualquer dos 3 ids, cached: true e latência <100ms. (c) Cronometrando a execução do demo-bonus.md em voz alta: sub 5 minutos, incluindo o ponto alto do eval report. (d) Ensaia com alguém que não conhece o projeto e pergunta se entendeu a separação extrator/narrador — se a resposta for "ah, o LLM não pode inventar número", o runbook tá claro.

# demo-bonus.md — Roteiro do endpoint explicável (5 min)

> Pré-requisito: `demo.md` da Fase 4 já ensaiado. Este aqui é o diferencial
> que você encaixa DEPOIS do beat 3 daquele runbook.

## Setup (60s antes da call)
```bash
./scripts/demo_bonus_warmup.sh https://credit-api-xxxxx-uc.a.run.app
```
Isso pré-aquece 3 ids (1 por tier) e abre: Swagger, Actions, e o último
`reports/eval_explainer_rule_*.json` no VS Code.

## Roteiro (5 min, 3 beats)

### Beat 1 · Decisão APROVADA (1 min)
- Swagger → `/explain-decision/rule/{id}` → `1234` → Execute.
- Aponta: `decision: APROVADO`, `risk_tier: LOW`, e a `narrative` em PT-BR
  citando os 3 fatores.
- Fala: "O tomador recebe decisão em segundos. Se o SaaS parceiro for
  mostrar numa tela, essa narrativa cumpre Res. 4.935/Bacen."

### Beat 2 · Decisão NEGADA (1.5 min)
- Swagger → `/explain-decision/9012` → Execute.
- Aponta: os `key_factors` têm `direction: alto_aumenta_risco`, e a
  `narrative` cita os números EXATOS (aponta com o cursor).
- Fala: "O LLM não pode inventar. O extrator determinístico escolheu os
  fatores, o LLM só narra. É isso que faz o compliance aprovar."

### Beat 3 · Killer — Eval como gate de CI (2 min)
- Alterna pra aba do VS Code com `reports/eval_explainer_2026XXXX.json`
- Aponta os campos na ordem:
  - `pass_rate_grounded: 1.0` → "nenhuma alucinação numérica em 21 amostras"
  - `pass_rate_forbidden: 1.0` → "nenhuma menção a CEP, raça, gênero"
  - `cost_usd_total: 0.0043` → "0.2 centavos por decisão"
  - `latency_p95_ms: 1847` → "sub 2 segundos"
- Fala: "Confiança em LLM em produção não vem de boa fé. Vem desse
  relatório. Ele roda no CI — se grounded-ness cai de 95%, o PR não passa."

## Perguntas prontas (específicas do LLM)

**"Como você sabe que o LLM não tá inventando número?"**
Abre o eval report, aponta `pass_rate_grounded`. O check é regex: todo
número na narrativa precisa existir entre os `value`/`median` dos
`key_factors`. 21 amostras, threshold de 95%, gate de CI.

**"Quanto isso custa?"**
`cost_usd_total: 0.0043` para 21 decisões — 0.2 centavos por chamada.
Haiku 4.5 pra LOW/MEDIUM, Sonnet só pra HIGH (variação). O `_usage` no
response carrega input/output tokens pra dashboard de custo.

**"E se a Anthropic cair?"**
Hoje o endpoint falha. Próximo passo: fallback retornando `decision` +
`key_factors` sem `narrative` — o extrator determinístico já é uma
explicação estruturada, só perde o wrapping em linguagem natural.
⚡ Variações deliberadas — escreva sua previsão antes de testar
  • Teste de alucinação manual: dentro de _build_prompt, remove a restrição "Não invente números" e passa um application_id com monthly_income extremo. O LLM arredonda, inventa comparações, ou respeita o valor? O que isso te diz sobre confiar em prompting sem eval automatizado?
  • Haiku vs Sonnet condicional: usa Haiku para tier LOW/MEDIUM e Sonnet só para HIGH (decisão de negativa é a mais sensível). Mede latência e custo por tier. O que justifica a escolha — qualidade da redação ou fidelidade aos fatos?
  • Guardrails explícitos na resposta: adiciona um campo guardrails_applied: list[str] listando as restrições do prompt. Útil quando o time de compliance perguntar "o que impede esse LLM de mencionar CEP?"
  • Duas audiências, um mesmo contexto: adiciona parâmetro ?audience=tomador|analista. Tomador recebe linguagem acessível em 2 frases; analista recebe jargão técnico com referência aos thresholds da política. Qual prompt muda, qual não?
  • Cache distribuído: quando o Cloud Run escala pra 5 instâncias, cada uma tem seu cache local. Qual é o cenário concreto em que isso vira problema? (dica: pense em reentreino do modelo + invalidação seletiva)
Teste da página em branco: descreve em voz alta para alguém do time de risco por que a separação entre extrator determinístico e narrador LLM é o que torna esse endpoint auditável. Se você consegue explicar em 30 segundos por que não dá pra passar o mart inteiro pro LLM e pedir "explica essa decisão" — você entendeu o núcleo.
