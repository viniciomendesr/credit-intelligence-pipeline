Desafio 4.1 — Logging estruturado com JSON · arquivo: src/logger.py
📥 Entrada: nome do módulo + run_id (str) 📤 Saída: logger configurado para emitir JSON por linha 🔗 Usado por: pipeline/flow.py — substitui todos os print()
🎯 O que construir

Projete get_logger(name, run_id=None) em src/logger.py — retorna um logger que emite JSON por linha, pronto para ingestão pelo Cloud Logging do GCP.

  • Formatter: JsonFormatter(fmt='%(asctime)s %(levelname)s %(message)s', rename_fields={'asctime': 'timestamp', 'levelname': 'level', 'message': 'event'}) — esses nomes são o que o Cloud Logging usa para indexação estruturada
  • Guard de handlers: if not logger.handlers: antes de adicionar o StreamHandler — sem isso, cada chamada de get_logger("pipeline") empilha um handler novo e cada evento é impresso N vezes
  • Em flow.py: RUN_ID = str(uuid.uuid4())[:8] gerado uma vez antes das tasks; passado em cada log via extra={"run_id": RUN_ID, "records": N} para rastreabilidade por execução
Por que JSON e não texto? Logs em texto são legíveis para humanos. Logs em JSON são legíveis para máquinas — e é o que o Cloud Logging do GCP espera para indexar e filtrar. Com um run_id único por execução (um uuid4()), você consegue filtrar todos os logs de uma execução específica com uma query simples.
📋 Guia passo a passo — abra só se travar
Passo 1 de 3 Configurar o formatter JSON com python-json-logger

Crie o handler com handler = logging.StreamHandler() e o formatter com jsonlogger.JsonFormatter(fmt='%(asctime)s %(name)s %(levelname)s %(message)s'). Use rename_fields={'asctime': 'timestamp', 'levelname': 'level', 'message': 'event'} para os nomes que o Cloud Logging do GCP espera. Antes de escrever: por que JSON em vez de texto? (logs em JSON são indexáveis e filtráveis por campo — texto livre não é)

Passo 2 de 3 Montar a função get_logger com guard de handlers

Envolva a criação do handler com if not logger.handlers: — sem esse guard, cada chamada de get_logger() adiciona um handler novo e cada evento é logado duplicado. Defina logger.setLevel(logging.INFO) antes de adicionar o handler. Antes de escrever: o que acontece sem o guard se get_logger("pipeline") for chamado em 3 tasks diferentes?

Passo 3 de 3 Usar run_id em cada chamada de log

Gere o run_id no início do flow com RUN_ID = str(uuid.uuid4())[:8] e passe para cada chamada de log: log.info("CSV ingerido", extra={"run_id": RUN_ID, "records": len(df)}). Com o run_id, você filtra todos os logs de uma execução específica no Cloud Logging com uma query simples. Antes de rodar: por que [:8]? UUIDs completos são longos demais para identificadores visuais em logs.

def get_logger(name: str, run_id: str = None) -> logging.Logger:
    logger = logging.getLogger(name)
    logger.setLevel(logging.INFO)
    if not logger.handlers:
        handler = logging.StreamHandler()

            rename_fields={
                'asctime': 'timestamp',
                'levelname': 'level',
                'message': 'event'
            }
        )
        handler.setFormatter(formatter)
        logger.addHandler(handler)
    return logger

Desafio 4.2 — Monitoramento e detecção de drift · arquivo: src/monitor.py
📥 Entrada: mart_credit_features.parquet + run_id 📤 Saída: data/monitoring/metrics_history.jsonl (append por execução) 🔗 Lido pelo: agente LLM do Bônus
🎯 O que construir

Projete em src/monitor.py três funções de observabilidade. Cada execução do pipeline adiciona uma linha ao arquivo JSONL — nunca sobrescreve.

  • collect_metrics(df, run_id): retorna dict com run_id, timestamp (ISO), total_records, default_rate_pct (mean × 100, round 4 — quatro casas para detectar variações pequenas), risk_tier_dist (value_counts normalize, round 2, to_dict), median_income (round 2)
  • save_metrics(metrics, path): open(path, 'a') + json.dumps(metrics) + '\n' — modo 'w' apagaria o histórico e o drift nunca seria detectado
  • detect_drift(path): compara lines[-2] e lines[-1] após json.loads(); retorna {"drift_detected": delta > 10.0, "delta_pct": round(delta, 4)} — threshold de 10 pontos percentuais em default_rate_pct
Por que JSONL e não um banco? JSONL (JSON Lines) é append-only — cada execução adiciona uma linha. É trivial de ler com Python, funciona sem setup, e é exatamente o que o agente LLM do módulo Bônus vai consumir para gerar análise em linguagem natural. Para escala maior você migraria para BigQuery, mas JSONL é o padrão correto para começar.
📋 Guia passo a passo — abra só se travar
Passo 1 de 3 Coletar métricas do DataFrame

Construa o dict de métricas com df['defaulted'].mean() * 100 para default rate e df['risk_tier'].value_counts(normalize=True).mul(100).round(2).to_dict() para distribuição. Inclua run_id e datetime.now().isoformat() para rastrear quando cada execução rodou. Antes de escrever: por que round(..., 4) e não 2 para o default rate? (4 casas permitem detectar variações menores — relevante para drift)

Passo 2 de 3 Persistir em JSONL com modo append

Salve com open(path, 'a') — modo append adiciona uma linha por execução sem sobrescrever. Escreva com f.write(json.dumps(metrics) + '\n'). Crie o diretório antes com Path(path).parent.mkdir(parents=True, exist_ok=True). Antes de escrever: o que acontece se usar open(path, 'w') por engano? (o histórico inteiro é apagado a cada execução — o drift nunca seria detectado)

Passo 3 de 3 Detectar drift comparando as duas últimas execuções

Leia o arquivo, separe por '\n', e compare lines[-2] com lines[-1] após json.loads(). Calcule delta = abs(curr['default_rate_pct'] - prev['default_rate_pct']) e retorne {"drift_detected": delta > threshold, "delta_pct": delta}. Antes de testar: edite manualmente a segunda linha do JSONL para ter default_rate_pct muito diferente e confirme que o drift é detectado.

import json
from datetime import datetime
from pathlib import Path
import pandas as pd

def collect_metrics(df: pd.DataFrame, run_id: str) -> dict:
    return {
        "run_id": run_id,
        "timestamp": datetime.().(),
        "default_rate_pct": round(df['defaulted'].() * 100, 4),
        "risk_tier_dist": df['risk_tier'].value_counts(
            normalize=True).mul(100).round(2).()
    }

def save_metrics(metrics: dict, path: str):
    Path(path).parent.mkdir(parents=True, exist_ok=True)
    with open(path, '') as f:  # append, não 'w'
        f.write(json.(metrics) + '\n')

def detect_drift(path: str) -> dict:
    lines = Path(path).read_text().strip().split('\n')
    if len(lines) < 2:
        return {"drift_detected": False}
    prev = json.loads(lines[])
    curr = json.loads(lines[])
    delta = abs(curr['default_rate_pct'] - prev['default_rate_pct'])
    return {
        "drift_detected": delta > ,
        "delta_pct": round(delta, 4)
    }

import json
from datetime import datetime
from pathlib import Path
import pandas as pd

def collect_metrics(df: pd.DataFrame, run_id: str) -> dict:
    return {
        "run_id": run_id,
        "timestamp": datetime.now().isoformat(),
        "total_records": len(df),
        "default_rate_pct": round(df['defaulted'].mean() * 100, 4),
        "risk_tier_dist": df['risk_tier'].value_counts(normalize=True).mul(100).round(2).to_dict(),
        "median_income": round(df['monthly_income'].median(), 2)
    }

def save_metrics(metrics: dict, path: str = 'data/monitoring/metrics_history.jsonl'):
    Path(path).parent.mkdir(parents=True, exist_ok=True)
    with open(path, 'a') as f:
        f.write(json.dumps(metrics) + '\n')

def detect_drift(path: str = 'data/monitoring/metrics_history.jsonl') -> dict:
    lines = Path(path).read_text().strip().split('\n')
    if len(lines) < 2:
        return {"drift_detected": False, "message": "histórico insuficiente"}
    prev = json.loads(lines[-2])
    curr = json.loads(lines[-1])
    delta = abs(curr['default_rate_pct'] - prev['default_rate_pct'])
    threshold = 10.0
    return {
        "drift_detected": delta > threshold,
        "delta_pct": round(delta, 4),
        "prev_run": prev['run_id'],
        "curr_run": curr['run_id'],
        "alert": f"Default rate mudou {delta:.2f}%" if delta > threshold else None
    }
Desafio 4.3 — Deploy no GCP Cloud Run · ferramenta: gcloud CLI
📥 Entrada: Dockerfile da Fase 3 + conta GCP 📤 Saída: URL pública (ex: https://credit-api-xxx.run.app) 🔗 Automatizado por: GitHub Actions no 4.4
🎯 O que construir

Faça o deploy da imagem do 3.3 no Cloud Run. A sequência tem quatro etapas em ordem — cada uma depende da anterior.

  • APIs: gcloud services enable run.googleapis.com artifactregistry.googleapis.com — sem isso, os comandos seguintes falham com erros de permissão ambíguos
  • Repositório: gcloud artifacts repositories create credit-api --repository-format=docker --location=us-central1
  • Build e push: IMAGE=us-central1-docker.pkg.dev/$PROJECT_ID/credit-api/credit-api:latest; gcloud auth configure-docker us-central1-docker.pkg.dev; docker build -t $IMAGE . && docker push $IMAGE
  • Deploy: gcloud run deploy credit-api --image $IMAGE --allow-unauthenticated --port 8000 --memory 512Mi — sem --port 8000, Cloud Run assume 8080 e a API não responde; guarde a URL gerada
Por que Cloud Run? É serverless — você não gerencia servidor, só o container. Escala a zero quando não tem tráfego (custo zero no free tier). Deploy é um comando. Para um projeto de portfólio, é exatamente o que você precisa: URL real, zero overhead operacional.
📋 Guia passo a passo — abra só se travar
Passo 1 de 3 Configurar projeto e criar repositório no Artifact Registry

Ative as APIs com gcloud services enable run.googleapis.com artifactregistry.googleapis.com, depois crie o repositório com gcloud artifacts repositories create credit-api --repository-format=docker --location=us-central1. O Artifact Registry é o registry privado do GCP — substitui o Docker Hub nesse contexto. Antes de rodar: por que usar Artifact Registry em vez de Docker Hub? (privacidade, integração com IAM do GCP, latência menor no deploy)

Passo 2 de 3 Build e push da imagem Docker

Configure o auth com gcloud auth configure-docker us-central1-docker.pkg.dev, depois: docker build -t $IMAGE . e docker push $IMAGE. A variável IMAGE deve seguir o padrão REGION-docker.pkg.dev/PROJECT_ID/REPO/SERVICE:TAG. Antes de rodar: se o build falhar, o push não acontece — corrija o Dockerfile antes de tentar o push.

Passo 3 de 3 Deploy no Cloud Run com URL pública

Rode gcloud run deploy credit-api --image $IMAGE --platform managed --region us-central1 --allow-unauthenticated --port 8000 --memory 512Mi. O --allow-unauthenticated torna a URL pública — sem ele, só requests autenticados com credenciais GCP funcionam. Após o deploy: guarda a URL gerada — você vai usá-la no CI/CD do 4.4 e pode mencioná-la na entrevista.

Desafio 4.4 — CI/CD com GitHub Actions · arquivo: .github/workflows/pipeline.yml
📥 Entrada: push no branch main 📤 Saída: testes rodados + nova versão deployada no Cloud Run automaticamente 🔗 Usa: Dockerfile (Fase 3) + credenciais GCP (GitHub Secrets)
🎯 O que construir

Projete em .github/workflows/pipeline.yml um workflow CI/CD com dois jobs em sequência. Trigger em push e pull_request para main.

  • Job test: ubuntu-latest, actions/setup-python@v5 com python-version: '3.11', pip install -r requirements.txt, pytest tests/ -v
  • Job deploy: needs: test + if: github.ref == 'refs/heads/main' — PRs ativam só o teste, nunca o deploy; autenticação via google-github-actions/auth@v2 com credentials_json: ${{ secrets.GCP_SA_KEY }}
  • Tag da imagem: $GITHUB_SHA — cada commit gera uma versão rastreável; qualquer revisão anterior é recuperável no Artifact Registry sem reconstrução
  • Secrets: GCP_SA_KEY e GCP_PROJECT_ID em Settings → Secrets → Actions — nunca em código; credenciais em código é o erro #1 de segurança em CI/CD
Onde ficam as credenciais do GCP? Em Settings → Secrets and variables → Actions do repositório GitHub — nunca em código. O workflow acessa via ${{ secrets.GCP_SA_KEY }}. Credenciais em código são o erro número 1 de segurança em pipelines de CI/CD.
📋 Guia passo a passo — abra só se travar
Passo 1 de 3 Criar o job de testes com dependências Python

No arquivo .github/workflows/pipeline.yml, crie o job test com runs-on: ubuntu-latest. Use as actions actions/checkout@v4 e actions/setup-python@v5 (com python-version: '3.11'), depois run: pip install -r requirements.txt e run: pytest tests/ -v. Antes de escrever: o que acontece se o requirements.txt estiver desatualizado e faltar um pacote? O job falha em pip install, não em pytest.

Passo 2 de 3 Criar o job de deploy com needs e condicional de branch

Crie o job deploy com needs: test (garante que testes passem antes) e if: github.ref == 'refs/heads/main' (PRs rodam só os testes, nunca o deploy). Use google-github-actions/auth@v2 com credentials_json: ${{ secrets.GCP_SA_KEY }}. Antes de configurar: adicione GCP_SA_KEY e GCP_PROJECT_ID em Settings → Secrets and variables → Actions — nunca em código.

Passo 3 de 3 Build, push e deploy no passo final do job

No step final do job deploy, use GITHUB_SHA como tag da imagem: IMAGE=$REGION-docker.pkg.dev/$PROJECT_ID/credit-api/$SERVICE:$GITHUB_SHA. Assim cada commit tem sua própria imagem — você consegue fazer rollback para qualquer versão anterior. Antes de rodar: commita um teste que falha de propósito e veja a aba "Actions" — o job de deploy não deve aparecer.

Desafio 4.5 — Demo ao vivo: sistema pronto pra ser demonstrado · arquivos: scripts/demo_warmup.sh + demo.md
📥 Entrada: URL pública do Cloud Run (gerada no 4.3) 📤 Saída: script de warm-up + runbook de 5 minutos que você segue pra qualquer demonstração do sistema rodando 🔗 Usa: /health + /risk-summary da API + GitHub Actions + Cloud Run Logs
🎯 O que construir

Dois artefatos pequenos mas críticos para transformar "tenho URL ao vivo" em "demo de 5 minutos que funciona em qualquer cenário de apresentação":

  • scripts/demo_warmup.sh: aceita $1 como URL do serviço, chama /health e /risk-summary sequencialmente, abre /docs no navegador. Rodado 60s antes de apresentar, garante que a instância do Cloud Run esteja quente (primeira chamada em cold start leva 3-5s — inaceitável ao vivo).
  • demo.md na raiz: runbook de 5 minutos estruturado em três beats — (1) "está no ar" via Swagger, (2) "qual é a carteira" via /risk-summary, (3) "como eu sei que não quebrou" mostrando a aba Actions do GitHub + logs do Cloud Run. Por que isso é um desafio e não uma nota de rodapé? #!/usr/bin/env bash
# scripts/demo_warmup.sh — aquece Cloud Run 60s antes de apresentar
set -euo pipefail

URL="${1:-https://credit-api-xxxxx-uc.a.run.app}"

echo "▶ Warm-up do Cloud Run: $URL"
echo ""
echo "1/2 · GET /health"
curl -s "$URL/health" | jq .
echo ""
echo "2/2 · GET /risk-summary (carrega o parquet)"
curl -s "$URL/risk-summary" | jq '{total_records, default_rate_pct, risk_tier_distribution_pct}'
echo ""

# Abre tabs que você vai mostrar
if command -v open >/dev/null; then
  open "$URL/docs"                               # Swagger
  open "https://github.com/SEU_USER/credit-intelligence-pipeline/actions"
  open "https://console.cloud.google.com/run"
else
  xdg-open "$URL/docs"
fi

echo "✅ Pronto. Espere 30s e comece a demo."
# demo.md — Runbook de demonstração do pipeline (5 min)

## Setup (60s antes da call)
```bash
./scripts/demo_warmup.sh https://credit-api-xxxxx-uc.a.run.app
```
Espere 30s. Isso aquece a instância e abre 3 tabs: Swagger, GitHub Actions, Cloud Run console.

## Roteiro (5 min, 3 beats)

### Beat 1 · Está no ar (60s)
- Abre Swagger `/docs`
- "A API expõe `/health` e `/risk-summary`. Deploy é no Cloud Run, URL pública."
- Clica `/risk-summary` → "Try it out" → Execute. Resposta aparece.
- "144 mil registros, default rate 6.8%, distribuição em 3 tiers. Dado real processado pelo pipeline."

### Beat 2 · Como o dado chegou aqui (2 min)
- "O mart é gerado por um pipeline Prefect: ingestão CSV + API de bureau → validação → DuckDB SQL → Parquet. A API não gera o dado, só serve. O parquet vem de um bucket GCS com SA dedicada."
- Se perguntarem sobre feature: "colunas são `risk_tier`, `default_rate`, `monthly_income`, etc — derivadas de um modelo SQL em `models/mart_credit_features.sql`."

### Beat 3 · Como sei que não quebrou (2 min)
- Abre aba GitHub Actions → último run verde.
- "Cada push pra main roda testes e faz deploy. Se teste falhar, deploy nem começa."
- Abre Cloud Run console → Logs.
- "Logging é JSON estruturado, indexável por `run_id`. Drift entre execuções é detectado e persistido em JSONL."

## Perguntas prontas

**"Como escala?"**
Cloud Run autoscale: min=0, max configurável. Escala a zero quando sem tráfego (custo zero). Primeira request depois de inatividade tem cold start de 2-3s — por isso o warm-up.

**"Como sabe que não quebrou em produção?"**
GitHub Actions bloqueia deploy se teste falhar. Cloud Run Logs capturam todo log JSON estruturado. Drift detection compara métricas entre runs do pipeline e persiste em `metrics_history.jsonl`.

**"Como faz rollback?"**
Cada deploy é uma Revisão no Cloud Run (tag = SHA do commit). `gcloud run services update-traffic credit-api --to-revisions REV=100` redireciona tráfego pra qualquer revisão anterior em segundos. Sem rebuild.
⚡ Variações deliberadas — escreva sua previsão antes de testar
  • Sobe o threshold de drift para 1% em vez de 10%. Quantas execuções idênticas ainda disparariam alerta? O que isso diz sobre calibrar limites de monitoramento?
  • Abre o Cloud Run console enquanto faz uma requisição. O que você vê em "Logs"? Compare com o output local.
  • Commita código que quebra um teste. O deploy anterior na URL pública continua funcionando enquanto o Actions falha?
  • Adiciona --min-instances 1 no deploy. O que muda no custo e na latência da primeira requisição (cold start)?
